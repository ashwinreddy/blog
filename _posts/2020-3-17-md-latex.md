I loathe googling a technical concept I used to know. It makes me feel like I never had ownership over it in the first place. Since sophomore year of high school, I've been looking for a robust way to build a [knowledge vault](https://www.calnewport.com/blog/2008/05/19/monday-master-class-how-to-build-a-knowledge-vault-and-avoid-wasting-an-entire-semesters-worth-of-work/) to track what I know more carefully. 

This blog post is _not_ about how to retain what you know. Instead, we're going to focus on the right tools for making the process of recording and retrieving that knowledge easy. These are lessons accumulated over a few years, so I hope you find them valuable. In addition, you'll find lots of links that can help you figure out what's best for your needs.

## Scope

Problems and solutions about technical knowledge come in different scopes. Here are three real problems I've had, and the kind of document I made to address the problem.

| Specific Problem I've Had                                    | General Solution to Class of Problem                         |
| ------------------------------------------------------------ | ------------------------------------------------------------ |
| I vaguely remember the connection between least squares as projection and as estimation. | A short document that explains one topic or focuses on the connection between two topics. |
| I've forgotten how to compute a matrix SVD.                  | An article in a wiki that explains the high level intuition and links to related ideas. |
| My friend wants to recall the big ideas from an information theory class we took together. | A self-produced [textbook](https://github.com/ashwinreddy/info-theory) of lecture notes to give me the details on a topic. |

Of these three, I've found the wiki the most eye-opening and helpful, and I would recommend it to anyone. A wiki makes it easy to start small and lets you scale up as you see fit. It also allows you to make connections between topics and then see those connections later when you need a refresher. A well-managed wiki is an excellent resource.

## Format

Despite the fact that I have three different forms of content, I'd like to be able to use the same format for all of them to simplify the process. The table below lists my requirements (ordered from broad to specific) that any potential format must meet.

| Requirement                    | Rationale                                                    | Eliminated Solution                                          |
| :------------------------------ | :------------------------------------------------------------: | ------------------------------------------------------------: |
| Digital.                       | No physical documents.                               | Can't use anything physical.                                 |
| Offline.                       | Easy to access in a pinch.                      | Can't use [Dropbox Paper](https://www.dropbox.com/paper) or [StackEdit](https://stackedit.io/). |
| PDF/HTML export. | Sharing/aesthetics.                                | Can't use a [plain text file](https://plaintextproject.online/). |
| LaTeX math.            | I write math fastest in LaTeX.[^game] |                                                              |
| Lightweight markup.   | Obtrusive syntax bad.                     | Can't use [MarkDeep](https://casual-effects.com/markdeep/) or [PubCSS](https://thomaspark.co/2015/01/pubcss-formatting-academic-publications-in-html-css/) or [LaTeX.css](https://latex.now.sh/) or LaTeX. If you're okay using raw LaTeX, try [this](https://castel.dev/post/lecture-notes-1/).[^tectonic] |

[^game]: I enjoy playing [typesetting games](https://texnique.xyz/)
[^tectonic]: The default TeX installation comes with lots of bloat I don't use and can be cryptic sometimes. I suggest using [tectonic](https://tectonic-typesetting.github.io/en-US/).

## Markdown

Two solutions that basically fit all the requirements are Markdown and Emacs' Org mode. Both are lightweight markup languages that typically support LaTeX. In theory, either one on their own could handle the three kinds of documents we care about. 

Personally, I'm a rather lazy person, and I prefer out of the box tools that just work. If you're not like me in this regard, [pandoc](https://pandoc.org/) can turn Markdown files and Org files into PDFs and HTML, and more [complicated setups](https://dave.autonoma.ca/blog/2019/05/22/typesetting-markdown-part-1/) exist for Markdown. Depending on context, I'd suggest [Typora](https://typora.io/) or [customizing Vim](https://secluded.site/vim-as-a-markdown-editor/) for Markdown, but really any text editing software works. Of course, Emacs is the best bet for Org mode, and I'd suggest the [spacemacs org layer](https://www.spacemacs.org/layers/+emacs/org/README.html). Org mode can take you far, and offers customization at scales both [big](https://lepisma.xyz/2017/10/28/ricing-org-mode/) and [small](https://sandyuraz.com/articles/orgmode-css/).



If you're just looking for Markdown tools that work, the table below tells you what tool I think is best for each kind of document.

| Type           | Markdown Tool                                                | Benefits                                                     |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| Short document | [Rmarkdown](https://rmarkdown.rstudio.com/) format and using the [RStudio](https://rstudio.com/) IDE. | Can include code + graphs.                                   |
| Wiki article   | [Gollum](https://github.com/gollum/gollum). My wiki is [public](https://github.com/ashwinreddy/wiki), if you need a starter. | Minimal but effective.                                       |
| Textbook       | [Bookdown](https://bookdown.org/home/). Here's my [template](https://github.com/ashwinreddy/rmarkdown-textbook). Or, you can just use [pandoc](https://keleshev.com/my-book-writing-setup/). | Handles definitions, theorems, and propositions out of the box. |

## Diagrams

This is the part I'm still working on, but [Inkscape](https://castel.dev/post/lecture-notes-2/#code) and [draw.io](https://draw.io/) or even its [VS Code extension](https://github.com/hediet/vscode-drawio) look like viable choices. If not, I have an iPad for hand-drawn diagrams, although I'm kind of fascinated by [computationally rendered](https://roughjs.com/) figures that look hand drawn. Based on the context, I might also use something like [Penrose](http://penrose.ink/).
